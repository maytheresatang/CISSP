Core Concepts Summary

SDLC Security Principle

Security is cheapest to fix at requirements/design phase. Cost to fix grows 10× per phase. Threat modelling → design phase. Penetration testing → pre-release.

Environment Separation

Dev → Test/QA → UAT → Production. Each must be separate. Prod data must NOT be used in Dev/Test. Developers should NOT have prod access.

OWASP Top 10 (Exam Focus)

Injection · Broken Auth · Sensitive Data Exposure · XXE · Broken Access Control · Security Misconfiguration · XSS · Insecure Deserialization · Vulnerable Components · Insufficient Logging

Testing Sequence

Unit → Integration → System → UAT → Regression. Security testing: SAST (static, early) → DAST (dynamic, running app) → IAST (runtime instrumentation).

DevOps → DevSecOps
Core Concept DevOps = Development + Operations (tear down the wall between them). DevSecOps = Dev + Sec + Ops — security is integrated throughout CI/CD pipeline, not bolted on at the end. Security shifts left (earlier in the lifecycle).
TermDefinitionExam Signal
CI/CDContinuous Integration / Continuous Delivery. Automated build, test, deploy pipeline.Security scans run automatically on each code commit.
Infrastructure as Code (IaC)Infrastructure defined in code (Terraform, Ansible). Version-controlled, repeatable.SDS (Software-Defined Security) = IaC applied to security controls.
Shift LeftMove security earlier in SDLC. Find defects at design/code, not in production.Cost to fix is lowest at design phase.
Software-Defined Security (SDS)Security infrastructure managed through code/APIs, not manual config.SDS is a type of IaC; DevSecOps context.
Container SecurityDocker/Kubernetes. Images should be minimal, scanned, immutable.Registry scanning, image signing (Notary).
Serverless SecurityFunction-as-a-Service. No OS to patch; risk is code and IAM permissions.Function permissions should be least privilege.

🪤 Exam Trap — CAB vs Change Management
Change Advisory Board (CAB) reviews and approves/rejects code changes in the release to production step. CAB does NOT write code, manage credentials, or prioritise development backlogs. If the question asks "who approves production code changes?" → CAB.

Secure Coding Principles
PrincipleWhat It Means
Input ValidationValidate ALL input on the server side. Never trust client-side validation alone. Prevents injection, XSS, buffer overflow.
Output EncodingEncode output before rendering to prevent XSS. Convert < > & to HTML entities.
Parameterised QueriesUse prepared statements for SQL. Prevents SQL injection. Escaping alone is insufficient.
Principle of Least PrivilegeCode runs with minimum permissions needed. DB accounts should be read-only where possible.
Error HandlingNever reveal stack traces or DB errors to end users. Log internally, show generic message externally.
Fail Secure (Fail Closed)On failure, default to DENY. Opposite of Fail Open (default allow) which is dangerous.
Defence in DepthMultiple layers: WAF → input validation → parameterised queries → least privilege DB account.
Separation of Duties in CodeDevelopers should not deploy to production. Reviewers should not merge their own code.
🔴 Critical — Client-side validation is NOT a security control Client-side input validation (JavaScript in browser) is BYPASSED easily by attackers. It improves UX only. Server-side validation is the required security control. Exam answer: client-side validation is NOT effective against SQL injection.

Security Testing Types — The Big Three
TypeWhat It DoesWhen RunExam Signal
SAST (Static Application Security Testing)Analyses source code WITHOUT running it. White-box by nature.During development, before compile"Code review tool", "source code scanner" → SAST
DAST (Dynamic Application Security Testing)Tests RUNNING application from outside. Black-box by nature.After deployment to test env"Fuzzing", "web app scanner", "Nikto", "Burp Suite" → DAST
IAST (Interactive AST)Agents inside running app. Combines SAST + DAST insight.During runtime testingMost accurate; high overhead.
RASP (Runtime Application Self-Protection)Security built into the app itself. Detects and blocks attacks at runtime.ProductionApp defends itself in real time.
⚠️ SAST vs DAST Key Distinction SAST = static, source code, early, white-box. DAST = dynamic, running app, later, black-box. Fuzzing is DAST (sends malformed inputs to running app). Code review is SAST (reading source). This is a high-frequency exam distinction.

Common Software Vulnerabilities
VulnerabilityHow It WorksMitigation
SQL InjectionMalicious SQL in input field manipulates query. Single quote (') is telltale sign.Parameterised queries, stored procedures, least-privilege DB account. Client-side validation is NOT effective.
Cross-Site Scripting (XSS)Attacker injects malicious script into web page viewed by victim.Output encoding, Content Security Policy (CSP), input validation.
Cross-Site Request Forgery (CSRF)Tricks authenticated user's browser into sending unauthorised request.CSRF tokens, SameSite cookie attribute, re-authentication for sensitive actions.
Buffer OverflowInput exceeds allocated memory, overwrites adjacent memory. Can execute arbitrary code.Input validation, bounds checking, safe languages (Rust, Java manage memory). Address Space Layout Randomization (ASLR).
Directory Traversal../../../etc/passwd — navigate outside web root using path sequences.Input validation, canonicalise paths, chroot jail.
Injection (general)SQL, LDAP, XML, OS command injection. Untrusted data sent to interpreter.Input validation, parameterisation, least privilege.
Insecure DeserialisationMalicious objects crafted in serialised data, executed on deserialisation.Integrity checks on serialised objects, type validation.
Session HijackingAttacker steals or predicts session token to impersonate user.TLS (most effective). Complex + expiring tokens (insufficient alone).
Logic BombMalicious code dormant until trigger condition met (e.g., fired employee)Code review, mandatory holidays, separation of duties.
Race Condition / TOCTOUTime-of-Check to Time-of-Use. State changes between check and use.Atomic operations, mutex locks, proper transaction handling.
Memory LeakApp fails to release unneeded memory. Resource exhaustion over time.Proper memory management, garbage collection, code review.
🔴 XSS Types — Know the Difference Persistent (Stored) XSS: Script stored in DB, served to every visitor. Example: forum post with embedded script. More dangerous.
Reflected (Non-persistent) XSS: Script in URL, reflected back in response. Victim must click malicious link.
Malware Relevant to Software Development
TypeDescriptionDev Context
Backdoor / TrapdoorHidden entry point left by developer. Bypasses normal authentication.Left in code intentionally or by mistake. Code review catches it.
Logic BombExecutes when condition met (date, event, login).Insider threat. Disgruntled developer may plant one.
Salami AttackSmall, unnoticeable fraud increments over time (e.g., rounding fractions to attacker account).Financial software vulnerability. Aggregate theft.
Covert ChannelExfiltrates data through unintended communication path (timing, storage).Hard to detect; bypasses normal access controls.
Database-Specific Vulnerabilities
IssueDescription
SQL InjectionSee above. Primary DB attack vector.
Inference AttackDeduce confidential data from non-confidential query results (statistical patterns).
Aggregation AttackCombining individually non-sensitive data items to reveal sensitive information.
Dirty ReadTransaction reads data written by another uncommitted transaction. Concurrency issue.
Lost UpdateTransaction overwrites data written by another transaction with earlier precedence.
Incorrect SummaryAggregate function runs while another transaction modifies the data. Produces wrong summary.
Database Security Concepts
ConceptDescription
Database ViewsRestrict what data users see. Virtual table showing subset of DB. Access control mechanism.
Stored ProceduresPre-compiled SQL stored in DB. Reduces injection risk; enforces consistent logic.
PolyinstantiationAllows different values for same data at different classification levels. MLS databases. Prevents inference.
Content-Dependent Access ControlAccess decisions based on data content (e.g., salary < £50k only). Row-level security.
Context-Dependent Access ControlAccess based on context (time, location, previous queries).
Database NormalisationEliminate data redundancy. 1NF → 2NF → 3NF → BCNF. Not a security control but reduces integrity risks.
ACID PropertiesAtomicity · Consistency · Isolation · Durability. Transaction integrity properties.
Mnemonic — ACID
"Always Check If Data survives"
Atomicity (all or nothing) · Consistency (valid state always) · Isolation (transactions don't interfere) · Durability (committed = permanent)
Database Types
TypeDescriptionSecurity Note
Relational (RDBMS)Tables, rows, SQL. Oracle, MS SQL, MySQL, PostgreSQL.Primary target for SQL injection. Use parameterised queries.
NoSQLDocument, key-value, graph, column-store. MongoDB, Cassandra, Redis.NoSQL injection exists. Input validation still required.
Object-Oriented DBObjects stored directly. Matches OOP languages.Access control must map to object methods.
Hierarchical DBTree structure. Parent-child relationships. IBM IMS.Legacy. Limited flexibility.
Data WarehouseAnalytical DB. Read-heavy. OLAP vs OLTP.Large data volumes = high sensitivity. Access controls critical.
⚠️ Aggregation vs Inference Inference: Deduce secret from non-secret query results. Example: average salary query reveals CEO pay by elimination.
Aggregation: Combine multiple non-sensitive pieces to reveal something sensitive. Example: name + address + employer + schedule = stalking risk.
Controls: Polyinstantiation, query result rounding, access restrictions on aggregate functions.
Acquired / Third-Party Software Security
CategoryDescriptionSecurity Concern
COTS (Commercial Off-the-Shelf)Vendor-built, commercially sold (MS Office, SAP)No source code access. Rely on vendor patches. Validate vendor security posture.
Open Source Software (OSS)Publicly available source code. Community maintained.Vulnerable components (OWASP #6). Must scan and maintain component inventory.
Third-Party Libraries / SDKsCode modules integrated into applications.Supply chain risk. Compromised library = compromised app (SolarWinds-type attack).
SaaS ApplicationsCloud-delivered software. Vendor manages infrastructure.Data residency, shared responsibility model, API security, SSO integration.
Managed Code vs UnmanagedManaged = runtime handles memory (.NET, Java). Unmanaged = manual memory (C, C++).Unmanaged code is more vulnerable to buffer overflows. Managed safer by design.
Software Bill of Materials (SBOM)
SBOM — Added to CISSP CBK 2024 An SBOM is a formal record of all components, libraries, and dependencies in a software product. It enables:
  • Rapid identification of affected systems when a vulnerable component (e.g., Log4j) is discovered
  • Supply chain risk management and vendor assurance
  • Regulatory compliance (US Executive Order 14028 mandates SBOMs for federal software)
  • End-of-Life tracking — know when components stop receiving patches
SBOM formats: SPDX (Linux Foundation), CycloneDX (OWASP), SWID (ISO).
Software Supply Chain Security
Attack TypeDescriptionControl
Dependency ConfusionAttacker publishes malicious package with same name as internal package in public registry.Use private registries, version pinning, integrity verification.
TyposquattingMalicious package with near-identical name (colour vs color).Verify package names and publishers.
Compromised UpdateLegitimate vendor update mechanism used to push malware (SolarWinds).Signed updates, hash verification, behaviour monitoring.
Malicious LibraryEmbedded malicious code in widely used OSS library (event-stream, xz-utils).SBOM, scanning, pinned versions, code review of dependencies.
Assessing Acquired Software Security
MethodDescription
Vendor AssessmentReview vendor security practices, SOC 2 reports, pen test results, patch history.
Software Composition Analysis (SCA)Automated tool scans codebase for known vulnerable open source components. Produces SBOM.
Binary AnalysisAnalyse compiled code (no source) for vulnerabilities. Used when source unavailable.

🪤 Exam Trap — EoL Software
End-of-Life software = no more patches from vendor. This is a significant risk. Controls: network isolation, compensating controls, virtualisation, migration planning. Exam will expect you to identify EoL as an unacceptable risk without compensating controls in high-sensitivity environments.
