Domain at a Glance

Domain 3 covers the engineering principles and models that underpin secure system design, plus cryptography and physical security. At 13% exam weight it is substantial. Cryptography and PKI are your identified weak areas — focus here first.

Core Security Design Principles
PrincipleDefinition
Least PrivilegeGrant only the minimum access required to perform a task.
Defence in DepthMultiple overlapping layers of controls — no single point of failure.
Fail Secure / Fail SafeFail Secure = deny access on failure. Fail Safe = allow access (life safety). Know which applies by context.
Separation of PrivilegeTwo conditions must be satisfied to grant access (two-person rule).
Economy of MechanismKeep design simple. Simple = easier to verify and test.
Open DesignSecurity should not rely on secrecy of design — only secrecy of keys (Kerckhoffs's Principle).
Complete MediationEvery access must be checked — no caching of permissions.
Least Common MechanismMinimise shared mechanisms — shared components create shared risk.

Bell-LaPadula vs Biba — Quick Rule
BLP = Read Down, Write Up (Confidentiality) · Biba = Read Up, Write Down (Integrity)
BLP protects SECRET data from leaking DOWN. Biba protects LOW integrity data from corrupting HIGH integrity data by flowing UP.
⚠️ Exam Trap: Clark-Wilson vs Bell-LaPadula Clark-Wilson uses well-formed transactions — data can only be modified through authorised procedures (TPs). This is the commercial/banking model. Bell-LaPadula is the military model focused purely on classification levels.
Common Evaluation Frameworks
FrameworkPurpose
Common Criteria (ISO 15408)International standard for security product evaluation. EAL 1–7 assurance levels. EAL 7 = highest (formal verification).
TCSEC (Orange Book)US DoD standard — D, C1, C2, B1, B2, B3, A1 levels. Largely superseded by Common Criteria.

Cryptography Fundamentals

Symmetric vs Asymmetric
SymmetricAsymmetric
KeysSame key for encrypt + decryptPublic key encrypts, private key decrypts
SpeedFast (1000× faster)Slow (computationally intensive)
Key distributionProblem — how to share the key securely?Solves key distribution (public key is public)
Scalabilityn(n-1)/2 keys for n parties2n keys for n parties
Use caseBulk data encryptionKey exchange, digital signatures, authentication
ExamplesAES, DES, 3DES, BlowfishRSA, ECC, Diffie-Hellman, ElGamal
📌 Hybrid Encryption (TLS model) Most real-world systems use both: asymmetric to exchange a symmetric session key securely, then symmetric for bulk data. TLS does this. RSA or ECC for the handshake, AES for the data stream.

Cryptographic Modes of Operation
ModeFull NameKey Feature
ECBElectronic CodebookEach block encrypted independently. Weak — identical plaintext = identical ciphertext. Do not use.
CBCCipher Block ChainingEach block XOR'd with previous ciphertext. Uses IV. Cannot parallelize. Common.
CTRCounterTurns block cipher into stream cipher. Parallelizable. No padding needed.
GCMGalois/Counter ModeCTR + authentication tag. Provides confidentiality AND integrity. AES-GCM = TLS 1.3 standard.
Cryptanalytic Attacks
AttackDescription
Birthday AttackExploits probability that two different inputs produce the same hash (collision). Targets hash functions.
Brute ForceTry every possible key. Defeated by long keys.
Dictionary AttackTry common words and phrases. Defeated by complex passwords + salting.
Rainbow TablePre-computed hash table. Defeated by salting hashes.
Known PlaintextAttacker has pairs of plaintext and ciphertext. Tries to derive key.
Chosen PlaintextAttacker can choose plaintext to encrypt and study output.
Meet in the MiddleAttacks double-encryption schemes like 2DES. Why 3DES was needed.
Side ChannelExploit physical implementation (power usage, timing, EMF). Not mathematical.
PKI Components
ComponentRole
CA (Certificate Authority)Issues and signs digital certificates. Root of trust. Root CA → Intermediate CA → End-entity cert.
RA (Registration Authority)Verifies identity before requesting a certificate from the CA. Offloads CA workload.
CRL (Certificate Revocation List)List of revoked certificates published by CA. Pull-based — client must download it.
OCSP (Online Certificate Status Protocol)Real-time certificate status check. Push-based — client queries responder. Faster than CRL.
Certificate RepositoryLDAP directory or database storing issued certificates.
X.509 Certificate Contents
  • Subject (owner of certificate)
  • Issuer (CA that signed it)
  • Serial number (unique per CA)
  • Validity period (Not Before / Not After)
  • Public key of the subject
  • Signature of the CA (proves authenticity)
  • Key usage extensions (what the cert can be used for)
Certificate Types
TypePurpose
DV (Domain Validated)Proves ownership of domain only. Automated. HTTPS padlock.
OV (Organisation Validated)CA verifies organisation exists. More trust than DV.
EV (Extended Validation)Rigorous vetting. Green bar in older browsers. Highest trust.
WildcardCovers a domain and all first-level subdomains (*.example.com).
SAN (Subject Alternative Name)Covers multiple domains in one certificate.
Code SigningSigns software to prove it hasn't been tampered with.
PKI Trust Models
ModelDescription
HierarchicalRoot CA → Intermediate CAs → End entities. Most common. Root CA kept offline.
Web of TrustPeers vouch for each other. Used in PGP. No central authority.
Bridge CAConnects separate PKI hierarchies. Used for cross-domain trust.
⚠️ Exam Trap: CRL vs OCSP CRL = list downloaded by client (pull). Can be stale between updates. OCSP = real-time query (push-style response). OCSP stapling = server includes OCSP response in TLS handshake, improving performance.
Physical Security Layers (Outside In)
  1. Perimeter — Fencing, walls, gates, vehicle barriers (bollards), CCTV, guard posts, lighting
  2. Facility / Building — Secured entrances, reception control, visitor management, mantraps/airlocks
  3. Internal Areas — Badge access, biometrics, CCTV, locked server rooms
  4. Server Room / Data Centre — Raised floors, hot/cold aisle containment, Faraday cage, biometric multi-factor entry
  5. Device Level — Cable locks, chassis locks, TPM, asset tags
Environmental Controls
ControlDetail
Temperature/HumidityData centres: 64–80°F (18–27°C), 40–60% relative humidity. Humidity too low → static. Too high → condensation.
Fire suppressionWet pipe (water — risk to equipment). Dry pipe (less risk). Halon replacement: FM-200, Inergen. CO2 (evacuate first — displaces oxygen).
HVACPositive pressure prevents contaminants from entering (outside air pressure kept lower).
UPS (Uninterruptible Power Supply)Battery backup for short outages. Buys time for generator startup.
GeneratorLong-term power backup. Needs fuel supply plan.
⚠️ Exam Trap: Fire Classes and Suppression Class A = ordinary combustibles (wood, paper) → water OK. Class B = flammable liquids → CO2, dry powder. Class C = electrical equipment → CO2, Halon replacement. Never use water on Class C (electrical shock risk).
Common Vulnerability Categories
Vulnerability TypeDescriptionMitigation
Buffer OverflowWriting beyond allocated memory. Can overwrite return addresses and execute arbitrary code.Input validation, ASLR, DEP/NX, safe coding (bounds checking)
Race Condition (TOCTOU)Time of Check to Time of Use — gap between checking and using a resource is exploited.Atomic operations, mutex locks
Covert ChannelCommunication through channels not designed for it (storage or timing channels).Bandwidth limitation, auditing
Backdoor / Maintenance HookHidden access mechanism left by developer. Major risk in third-party code.Code review, static analysis
Privileged EscalationGaining higher permissions than authorised.Least privilege, OS hardening
RootkitHides malware by subverting OS kernel. Extremely difficult to detect.Integrity monitoring, Secure Boot, TPM
BootkitInfects MBR/UEFI — loads before OS. TPM + Secure Boot specifically designed to prevent this.TPM, Secure Boot, measured boot
🚨 Quiz Gap: Bootkit vs Rootkit Both hide at deep system levels. Bootkit = infects bootloader/MBR/UEFI, runs before OS. Rootkit = infects OS kernel, runs within OS. TPM and Secure Boot are designed specifically against bootkits. The exam will try to make you pick "rootkit."
