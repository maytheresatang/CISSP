⚠️ Exam Trap: Identification vs Authentication Presenting your username card = identification. Entering your PIN = authentication. Identification is claiming who you are; authentication is proving it. They are always separate steps.

Authentication Factors
FactorCategoryExamplesWeakness
Type 1Something you knowPassword, PIN, passphrase, security questionCan be shared, guessed, phished
Type 2Something you haveHardware token (RSA SecurID), smart card, OTP via phone, FIDO2 keyCan be lost or stolen

📌 MFA requirement MFA = at least two different factor types. Password + PIN = 1FA (both Type 1). Password + token = MFA (Type 1 + Type 2).
Biometric Error Rates
MetricFull NameDefinitionExam Relevance
FARFalse Acceptance Rate% of unauthorised users incorrectly accepted. Type II error. Security risk.Lower FAR = more secure
FRRFalse Rejection Rate% of authorised users incorrectly rejected. Type I error. Usability issue.Lower FRR = more usable
CERCrossover Error RatePoint where FAR = FRR. Best overall accuracy measure.Lower CER = better system
⚠️ FAR vs FRR trade-off You cannot simultaneously minimise both. Increasing sensitivity (accept fewer people) → lower FAR (more secure) but higher FRR (more false rejections). The CER is the "sweet spot." High security = lower FAR (accept fewer). High convenience = lower FRR (reject fewer legit users).
Password Policies (NIST SP 800-63B)
  • Minimum 8 characters (15+ recommended for admin accounts)
  • Check against breached password lists
  • Do NOT force periodic password changes without evidence of compromise
  • Do NOT require complexity rules that lead to predictable patterns (P@ssw0rd!)
  • Use passphrases — length beats complexity
  • Support password managers

SAML vs OAuth vs OpenID Connect — Critical Distinction
ProtocolPurposeToken TypeBest Use Case
SAML 2.0Authentication AND AuthorisationXML assertionsEnterprise SSO (corporate portals, federated login between orgs)
OAuth 2.0Authorisation ONLY — not authenticationAccess tokenAPI access delegation (app accessing your Google Drive)
OpenID Connect (OIDC)Authentication (built on OAuth 2.0)ID token (JWT)"Sign in with Google/Apple" — consumer identity
🚨 Exam Trap: OAuth is NOT authentication OAuth 2.0 grants an app permission to act on your behalf — it does NOT authenticate you. OIDC adds the identity layer on top of OAuth. SAML is typically enterprise-to-enterprise. OIDC is consumer/cloud-to-cloud.
Federation Mnemonic
SAML = Enterprise SSO · OAuth = API delegation · OIDC = "Sign in with..."
Scenario: company merger → SAML. App accessing your calendar → OAuth. "Login with Google" → OIDC.
Federation Roles

JIT (Just-In-Time) Provisioning

User accounts in the SP are created automatically on first login using IdP assertion data. No pre-provisioning needed. Common in cloud SaaS federated environments.

DiscretionaryDACData ownerOwner sets permissions on their objects. Flexible but weak — permission creep risk.Windows file system
MandatoryMACSystem / policyLabels on subjects and objects. No owner override. Most secure. Bell-LaPadula model.Military classification

⚠️ RBAC vs ABAC RBAC: "What is your role?" ABAC: "What are ALL your attributes, and what context does this request occur in?" ABAC is more granular — can evaluate time of day, device health, location. Zero Trust implementations use ABAC-style continuous evaluation.

Identity Provisioning Lifecycle
  1. Provision — Create account, assign minimum role, grant baseline access
  2. Maintain — Update roles on job change, add access by exception with approval
  3. Review — Regular access certification / recertification (user access reviews)
  4. Deprovision — Disable account immediately on termination, revoke tokens, collect hardware
🚨 Deprovisioning is a critical control Terminated employee accounts left active are a leading insider threat vector. The CISSP answer for "most important action upon termination" is immediately disable/revoke access — not the exit interview, not collecting equipment.

PAM — Privileged Access Management

PAM controls and monitors accounts with elevated privileges (admins, service accounts, root).

  • Just-in-time (JIT) access — Elevate privilege only when needed, auto-expire
  • Session recording — All privileged session activity logged and reviewable
  • Password vaulting — Privileged passwords managed by PAM system (CyberArk, BeyondTrust)
  • Break-glass accounts — Emergency admin accounts; access triggers alert and full audit
Zero Trust Architecture (NIST SP 800-207)

Zero Trust = "never trust, always verify." No implicit trust based on network location. Every request is verified as if it originates from an untrusted network.

Zero Trust Core Principles
  • Verify explicitly — always authenticate and authorise based on all available data points
  • Use least privilege access — limit user access with JIT, JEA, risk-based adaptive policies
  • Assume breach — minimise blast radius, segment access, encrypt everything, use analytics
Zero Trust Components (NIST 800-207)
ComponentRole
Policy Decision Point (PDP)Makes access decisions. Contains Policy Engine (PE) + Policy Administrator (PA).
Policy Enforcement Point (PEP)Enforces decisions made by PDP. Gate between subject and resource.
Policy Engine (PE)Evaluates trust score, applies policies, grants or denies.
Policy Administrator (PA)Signals PEP to allow or deny; issues session tokens.
SASE (Secure Access Service Edge)

SASE = SD-WAN + cloud-native security (CASB, FWaaS, ZTNA, SWG) delivered as a single cloud service. Eliminates the concept of a network perimeter — security travels with the user.

⚠️ ZTNA vs VPN VPN gives network-level access (once connected, trust is implicit). ZTNA gives application-level access only — each application is a separate policy decision. ZTNA = Zero Trust Network Access.
